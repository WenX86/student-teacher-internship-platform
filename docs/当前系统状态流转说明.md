# 当前系统状态流转说明

## 1. 文档目的

- 本文档用于说明当前代码中已经实际实现的状态流转规则。
- 文档以 `backend-spring` 当前实现为准，适合作为后续补测试、排查流程问题和继续开发的依据。
- 若需求文档中存在更复杂的状态设计，而当前代码尚未落地，本文会单独标注为“未实现”。

## 2. 适用范围

本文覆盖以下几类核心对象：

- 指导教师申请
- 实习申请
- 过程表单
- 学院入驻申请
- 账号状态
- 补充状态：学生实习状态、评价提交流转标记

## 3. 角色说明

- 学生：发起指导教师申请、实习申请、过程表单提交
- 指导教师：确认或驳回指导教师申请、审核过程表单、填写评价
- 学院管理员：复核指导关系、审批实习申请、终审或归档表单、确认评价
- 超级管理员：审核学院入驻申请、管理学院管理员账号状态

## 4. 指导教师申请状态

### 4.1 状态集合

- `待教师确认`
- `教师驳回`
- `待学院复核`
- `学院驳回`
- `已生效`

### 4.2 流转规则

1. 学生发起指导教师申请后，状态进入 `待教师确认`
2. 指导教师同意后，状态进入 `待学院复核`
3. 指导教师驳回后，状态进入 `教师驳回`
4. 学院管理员复核通过后，状态进入 `已生效`
5. 学院管理员复核不通过后，状态进入 `学院驳回`

### 4.3 触发角色与动作

- 学生：
  - 可发起指导教师申请
- 指导教师：
  - 可处理发给自己的申请
  - 动作为“确认”或“驳回”
- 学院管理员：
  - 可复核指导教师已确认的申请
  - 动作为“复核通过”或“复核驳回”

### 4.4 前置条件与限制

- 教师只能处理分配给自己的指导申请
- 学院管理员当前实现中按申请直接复核，没有额外的“仅允许待学院复核时操作”的显式拦截逻辑写在方法里
- 实习申请创建前，学生必须至少存在一条 `已生效` 的指导关系

### 4.5 关键副作用

- 教师确认后，会给学生发送处理结果消息
- 教师确认通过后，会给学院管理员发送“待复核”提醒
- 学院复核后，会给学生发送正式结果消息
- 学院复核通过时，会写入 `effectiveAt`

## 5. 实习申请状态

### 5.1 状态集合

- `待学院审批`
- `已退回`
- `已通过`

### 5.2 流转规则

1. 学生提交实习申请后，状态进入 `待学院审批`
2. 学院管理员审批通过后，状态进入 `已通过`
3. 学院管理员审批驳回后，状态进入 `已退回`

### 5.3 触发角色与动作

- 学生：
  - 创建实习申请
- 学院管理员：
  - 审批实习申请
  - 可登记单位确认结果、反馈意见、接收时间、审批意见

### 5.4 前置条件与限制

- 学生提交实习申请前，必须存在 `已生效` 的指导教师关系
- 当前实现中，学生端是“新增申请”模式，没有明确的“修改后再次提交”独立状态链

### 5.5 关键副作用

- 申请创建后，会向学院管理员发送待办提醒
- 审批通过后，学生的 `internshipStatus` 会更新为 `实习中`
- 审批结果会发送给学生

## 6. 过程表单状态

### 6.1 当前代码已实现状态

- `草稿`
- `教师审核中`
- `教师退回`
- `学院审核中`
- `学院退回`
- `已归档`

### 6.2 流转规则

1. 学生新建表单但不提交时，状态为 `草稿`
2. 学生新建或编辑表单并提交时，状态为 `教师审核中`
3. 教师审核通过后，状态进入 `学院审核中`
4. 教师审核退回后，状态进入 `教师退回`
5. 学院审核通过后，状态进入 `已归档`
6. 学院审核退回后，状态进入 `学院退回`
7. 学生在已有表单上再次编辑：
   - 保存草稿时，重新回到 `草稿`
   - 提交时，重新进入 `教师审核中`

### 6.3 触发角色与动作

- 学生：
  - 新建表单
  - 编辑自己的表单
  - 保存草稿
  - 提交审核
- 指导教师：
  - 审核自己负责学生的表单
  - 动作为“通过”或“退回”
- 学院管理员：
  - 审核本学院学生表单
  - 动作为“归档”或“退回”
  - 支持批量归档/批量退回

### 6.4 前置条件与限制

- 学生保存或编辑表单前，模板必须处于启用状态
- 教师只能审核当前对该学生有效的指导教师关系对应的表单
- 学院管理员只能处理本学院学生的表单
- 批量归档或批量退回只允许处理状态为 `学院审核中` 的表单

### 6.5 关键副作用

- 学生提交后，会向指导教师发送待审核提醒
- 教师审核通过后，会向学院管理员发送待归档提醒
- 教师审核结果会通知学生
- 学院审核结果会通知学生
- 学生编辑已有表单时，会写入历史版本 `historyJson`

### 6.6 与需求文档的差异说明

需求分析中出现过以下状态或概念，但当前代码尚未作为独立状态实现：

- `已提交`
- `教师通过`
- `修改申请中`
- `允许修改`
- `已重新提交`

当前实现采用的是更简化的状态链：

- 学生一旦提交，直接进入 `教师审核中`
- 退回后学生重新编辑，再次提交时直接回到 `教师审核中`
- 当前没有独立的“归档后申请修改”状态流程

## 7. 学院入驻申请状态

### 7.1 状态集合

- `待审核`
- `已通过`
- `已驳回`

### 7.2 流转规则

1. 学院提交入驻申请后，状态进入 `待审核`
2. 超级管理员审核通过后，状态进入 `已通过`
3. 超级管理员审核驳回后，状态进入 `已驳回`

### 7.3 触发角色与动作

- 申请方：
  - 提交入驻申请
- 超级管理员：
  - 审核通过
  - 审核驳回

### 7.4 关键副作用

审核通过时：

- 系统按“学校名称 + 学院名称”检查学院基础档案
- 若学院不存在则创建，存在则补充联系人等信息
- 若该学院尚无学院管理员账号，则自动创建账号
- 审核结果会在返回值和 `reviewComment` 中回显账号信息
- 写入审计日志

审核驳回时：

- 更新驳回状态
- 保留审核意见
- 写入审计日志

## 8. 账号状态

### 8.1 通用账号状态

当前系统实际可见的账号状态主要为：

- `ACTIVE`
- `DISABLED`

登录规则：

- 只有 `ACTIVE` 账号可以正常登录
- `DISABLED` 账号会在登录时被判定为不可用

### 8.2 学生账号

支持操作：

- 创建账号：默认 `ACTIVE`
- 重置密码：密码重置为 `123456`，并将 `mustChangePassword` 设为 `true`
- 修改账号状态：当前实现允许学院管理员直接更新账号状态值

说明：

- 学生扩展表中的 `internshipStatus` 与 `user_account.status` 不是同一个概念

### 8.3 教师账号

支持操作：

- 创建账号：默认 `ACTIVE`
- 重置密码：密码重置为 `123456`，并将 `mustChangePassword` 设为 `true`
- 启用/停用：当前主要使用 `ACTIVE` / `DISABLED`

说明：

- 教师账号状态会同时更新 `user_account.status` 与 `teacher.status`

### 8.4 学院管理员账号

支持操作：

- 审核入驻时自动创建：默认 `ACTIVE`
- 重置密码：密码重置为 `123456`，并将 `mustChangePassword` 设为 `true`
- 启用/停用：仅允许 `ACTIVE` 或 `DISABLED`

说明：

- 学院管理员账号创建后默认要求首次登录修改密码
- 超级管理员端可直接管理学院管理员账号状态

### 8.5 首次改密标记

- 新创建账号默认 `mustChangePassword = true`
- 管理员重置密码后，`mustChangePassword` 重新设为 `true`
- 用户主动修改密码后，`mustChangePassword = false`

## 9. 学生实习状态

### 9.1 当前已实现状态

- `待申请`
- `实习中`

### 9.2 流转规则

1. 新建学生账号时，初始状态为 `待申请`
2. 实习申请经学院审批通过后，状态变为 `实习中`

### 9.3 当前限制

- 当前代码中未实现更细的学生实习阶段状态，例如“待审核”“已完成”“已终止”等
- 若实习申请被退回，不会自动回写一个新的学生实习状态

## 10. 评价确认标记

评价记录当前没有单独的字符串状态字段，而是采用两个布尔标记表达流程进度：

- `submittedToCollege`
- `confirmedByCollege`

### 10.1 流转规则

1. 教师首次保存或更新评价后：
   - `submittedToCollege = true`
   - `confirmedByCollege = false`
2. 学院管理员确认评价后：
   - `confirmedByCollege = true`
   - 写入 `collegeConfirmedAt`

### 10.2 触发角色

- 指导教师：填写并提交评价
- 学院管理员：确认评价并可给出学院最终成绩

## 11. 当前开发建议

基于现有实现，后续开发建议优先处理以下事项：

1. 为各状态流转补充集成测试，尤其是“非法状态跳转”的拦截测试
2. 若要支持需求文档中的“修改申请中 / 允许修改 / 已重新提交”，需先扩展表单状态模型
3. 为学生和教师账号状态增加更严格的可选值校验，避免直接写入任意字符串
4. 若要细化学生实习生命周期，建议将 `internshipStatus` 独立成明确的状态机
