# 师范生教育实习全过程管理平台软件详细设计说明书

## 1. 引言

### 1.1 编写目的

本文档依据《需求规格说明书V2》编制，用于说明师范生教育实习全过程管理平台的软件详细设计方案，作为系统开发、联调测试、部署实施、运维交接和后续迭代的依据。文档重点覆盖系统概要设计、模块设计、数据库设计、接口设计、界面设计、安全设计及非功能实现方案。

### 1.2 适用范围

本说明书适用于师范生教育实习全过程管理平台一期建设范围，覆盖学生端、指导教师端、学院管理员端、超级管理员端，以及与之配套的认证授权、流程审批、消息通知、附件管理、日志审计、系统配置等支撑能力。

### 1.3 设计依据

1. 《需求规格说明书V2》
2. 《师范生教育实习全过程管理平台-需求分析》
3. 当前项目一期实现方案与代码结构
4. 高校教育实习业务管理通用规范

### 1.4 术语说明

1. 学生：参与教育实习并提交过程材料的师范生用户。
2. 指导教师：负责指导学生、审核过程材料并提交评价的教师用户。
3. 学院管理员：负责学院基础数据维护、过程审批、终审归档与统计分析的管理用户。
4. 超级管理员：作为学校总管理员，负责本校基础数据、学院管理、参数配置和日志审计。
5. 表单模板：由平台配置的、用于生成具体业务表单的数据结构定义。
6. 表单实例：学生基于模板填写并提交的实际业务记录。

## 2. 系统概要设计

### 2.1 建设目标

平台面向师范生教育实习全过程管理场景，建设统一的数字化业务平台，实现实习申请、指导教师申请、过程填报、过程审核、指导记录、评价确认、材料归档、统计分析和消息提醒的一体化管理。系统应解决传统线下材料分散、审核链条不透明、数据难沉淀、过程监管弱等问题。

### 2.2 设计原则

1. 业务闭环原则。围绕“申请、审核、记录、评价、归档、统计”构建完整流程。
2. 分级授权原则。按角色进行功能边界划分，保证最小权限访问。
3. 配置驱动原则。表单模板、系统参数、预警阈值等支持后台配置。
4. 可追溯原则。关键审批、状态流转、消息提醒和操作行为可查询、可审计。
5. 可扩展原则。为后续二期、三期扩展实习阶段管理、移动端能力、数据分析等预留空间。

### 2.3 总体技术架构

系统采用前后端分离的 B/S 架构，由前端展示层、后端业务层、数据存储层和文件存储层构成。

1. 前端展示层采用 Vue 3 + Vite + Vue Router + Pinia + Element Plus，负责页面渲染、状态管理、交互校验和接口调用。
2. 后端业务层采用 Spring Boot + Spring Security + MyBatis-Plus，负责认证授权、业务编排、数据校验、状态流转和审计记录。
3. 数据存储层采用 MySQL，开发环境可兼容 H2 内存库，负责业务主数据、流程数据、配置数据和日志数据持久化。
4. 文件存储层采用本地文件存储目录，负责附件上传、访问和下载。

### 2.4 系统逻辑架构

系统逻辑上分为四层。

1. 表现层：登录页、门户布局、学生端页面、教师端页面、学院端页面、平台管理页。
2. 接口层：认证接口、业务接口、文件接口、统计接口、配置接口。
3. 服务层：流程服务、账号服务、消息服务、预警服务、统计服务、文件服务。
4. 数据层：业务表、流程表、消息表、日志表、参数表、模板表。

### 2.5 部署架构

一期建议采用单体部署方案。

1. 前端应用打包为静态资源，部署在 Nginx 或学校统一 Web 服务环境中。
2. 后端应用以 Spring Boot Jar 方式部署在应用服务器。
3. MySQL 数据库独立部署，建议与应用服务分离。
4. 附件上传目录部署在应用服务器独立磁盘或共享文件目录。
5. 日志文件与数据库备份分别存储，避免与业务附件混用。

### 2.6 角色与权限架构

平台采用 RBAC 思路进行授权，角色包括学生、指导教师、学院管理员、超级管理员。

1. 学生拥有个人资料维护、指导教师申请、实习申请、表单填报、消息查看、结果查看等权限。
2. 指导教师拥有指导关系确认、过程材料审核、指导记录维护、评价填写、提醒督办查看等权限。
3. 学院管理员拥有学生管理、教师管理、实习单位管理、实习审批、归档审批、评价确认、统计报表、预警查看等权限。
4. 超级管理员拥有学院管理、学院管理员账号开通、基础数据管理、系统参数管理、表单模板管理、日志审计等权限。

### 2.7 模块结构设计

系统按业务域划分为以下核心模块。

1. 认证与账号管理模块
2. 学生业务模块
3. 指导教师业务模块
4. 学院管理模块
5. 超级管理模块
6. 流程与审核模块
7. 表单模板与归档模块
8. 消息通知与预警模块
9. 附件与文件管理模块
10. 统计分析与审计模块

## 3. 模块详细设计

### 3.1 认证与账号管理模块

该模块负责登录认证、令牌签发、密码修改、首次登录改密、账户启停用和个人身份信息获取。

1. 用户通过账号密码登录系统。
2. 后端校验账户状态、密码摘要和角色信息，校验通过后生成令牌。
3. 前端将令牌和用户信息写入本地存储，并根据角色跳转到对应门户。
4. 用户首次登录或管理员重置密码后，系统通过 `mustChangePassword` 标识强制修改密码。
5. 账户状态统一采用 `ACTIVE` 与 `DISABLED` 控制登录能力。

### 3.2 学生业务模块

学生模块负责学生在实习全过程中的自助填报与查询。

1. 个人中心子模块：维护基础信息、联系方式和实习类型。
2. 指导教师申请子模块：向教师发起指导申请，跟踪审核结果。
3. 实习申请子模块：提交实习单位、岗位、时间和目标等申请信息。
4. 过程表单子模块：基于模板新建、保存、提交、修改和查看表单实例。
5. 材料消息子模块：查看待办、退回原因、审核结果与归档结果。
6. 结果查看子模块：查看评价结果和归档状态。

### 3.3 指导教师业务模块

教师模块负责过程指导和过程审核。

1. 指导申请处理子模块：确认或驳回学生发起的指导教师申请。
2. 材料审核子模块：审核学生提交的过程表单并给出意见和评分。
3. 指导记录子模块：记录指导时间、方式、问题、建议和后续跟进。
4. 评价管理子模块：填写阶段评价、总结评价、终评成绩，并提交学院确认。
5. 提醒督办子模块：查看超时待审、待跟进和提醒信息。

### 3.4 学院管理模块

学院管理员模块是业务闭环的核心管理端。

1. 学生管理子模块：新增学生、维护学生资料、重置密码、启停账户。
2. 教师管理子模块：新增教师、维护教师资料、统计指导人数、启停账户。
3. 实习单位管理子模块：维护单位名称、地址、联系人、合作状态等信息。
4. 指导关系复核子模块：复核教师已确认的指导申请并使其生效。
5. 实习申请审批子模块：审批学生实习申请，记录单位确认结果与审批意见。
6. 归档中心子模块：终审或批量归档学生表单，处理退回。
7. 评价确认子模块：确认教师提交的评价结果，补充学院意见与分数。
8. 报表分析子模块：按学生、教师、学院维度查看汇总数据。
9. 预警督办子模块：查看超时未审、超时未改和待确认预警，发送督办消息。

### 3.5 超级管理模块

超级管理员面向本校范围内的平台治理与学院管理。

1. 学院管理子模块：维护学院信息，并为各学院开通学院管理员账户。
2. 学院与学院管理员管理子模块：管理学院、角色、字典项等平台基础信息。
3. 表单模板管理子模块：维护模板编码、适用类型、字段结构和启用状态。
4. 系统参数管理子模块：维护超时天数、预警等级、提醒模板等配置。
5. 日志审计子模块：查看登录日志、审核动作和系统操作痕迹。

## 4. 业务流程与状态机设计

### 4.1 指导教师申请流程

流程状态包括“待教师确认、教师驳回、待学院复核、学院驳回、已生效”。

1. 学生提交申请后进入待教师确认。
2. 教师同意后进入待学院复核，驳回则流程结束。
3. 学院管理员复核通过后状态变为已生效。
4. 已生效的指导关系作为后续实习申请和材料审核的前置条件。

### 4.2 实习申请流程

流程状态包括“待学院审批、已通过、已退回”。

1. 学生只有在指导关系生效后才能提交实习申请。
2. 学院管理员审批通过后，学生实习状态更新为“实习中”。
3. 若审批退回，学生需重新提交新的申请记录。

### 4.3 表单提交流程

流程状态包括“草稿、教师审核中、教师退回、学院审核中、学院退回、已归档”。

1. 学生新建表单可先保存为草稿。
2. 提交后进入教师审核中。
3. 教师审核通过后进入学院审核中，退回则进入教师退回。
4. 学院审核通过后进入已归档，退回则进入学院退回。
5. 学生对退回表单再次修改提交时，版本号递增并记录历史。

### 4.4 评价确认流程

评价流程采用标记位表示。

1. 教师提交评价后，`submittedToCollege = true`。
2. 学院确认后，`confirmedByCollege = true`，并记录确认时间。
3. 已确认评价不允许教师继续修改。

### 4.5 消息通知流程

关键节点触发消息通知。

1. 发起指导教师申请时通知教师。
2. 教师确认后通知学院管理员。
3. 实习申请提交后通知学院管理员。
4. 表单提交后通知指导教师。
5. 教师审核通过后通知学院管理员。
6. 教师退回或学院退回后通知学生。
7. 评价提交后通知学院管理员确认。

## 5. 数据库详细设计

### 5.1 数据库选型与原则

一期数据库选用 MySQL，开发联调可使用 H2 兼容模式。设计遵循以下原则。

1. 业务主实体独立建表。
2. 审批流转与主业务分离存储，但保留核心状态字段。
3. 模板、附件、历史版本、评分维度等可变结构采用 JSON 字段扩展。
4. 关键时间节点统一保留时间戳，支持追踪与统计。

### 5.2 核心实体关系设计

1. `college` 与 `user_account` 为一对多关系。
2. `user_account` 与 `student`、`teacher` 为一对一扩展关系。
3. `student` 与 `mentor_application`、`internship_application`、`form_instance`、`evaluation_record` 为一对多关系。
4. `teacher` 与 `mentor_application`、`guidance_record`、`evaluation_record` 为一对多关系。
5. `organization` 与 `internship_application` 为一对多关系。
6. `form_template` 与 `form_instance` 为一对多关系。
7. `user_account` 与 `message_notice`、`audit_log` 为一对多关系。

### 5.3 数据表设计说明

#### 5.3.1 学院表 `college`

用于保存学院主数据，包括学校名称、学院名称、联系人、联系电话和简介信息。该表是学院管理员、教师、学生的数据归属基础。

#### 5.3.2 用户账户表 `user_account`

用于保存登录账号、姓名、角色、密码摘要、首次改密标志、账户状态、所属学院和最近登录时间。该表是统一身份认证的核心表。

#### 5.3.3 教师表 `teacher`

用于保存教师业务信息，包括工号、院系、电话、所属学院和业务状态。通过 `user_id` 与账户表关联。

#### 5.3.4 学生表 `student`

用于保存学生业务信息，包括学号、专业、班级、实习类型、实习状态、联系电话和资料完整标志。通过 `user_id` 与账户表关联。

#### 5.3.5 实习单位表 `organization`

用于保存学院维护的实习单位信息，包括地址、联系人、联系电话、单位性质和合作状态。

#### 5.3.6 指导教师申请表 `mentor_application`

用于记录学生与教师之间的指导申请流程数据，包括申请状态、双方意见、学院意见和关键审核时间。

#### 5.3.7 实习申请表 `internship_application`

用于记录学生实习申请，包括实习单位、批次、岗位、成绩目标、起止时间、备注、附件、单位确认情况和审批意见。

#### 5.3.8 表单模板表 `form_template`

用于保存模板元数据，包括模板编码、模板名称、分类、描述、适用实习类型、字段结构 JSON、启停状态和排序号。

#### 5.3.9 表单实例表 `form_instance`

用于保存学生填报的过程材料，包括模板信息、当前状态、版本号、内容 JSON、附件 JSON、教师意见、学院意见、评分、提交时间、审核时间和历史版本 JSON。

#### 5.3.10 消息通知表 `message_notice`

用于保存系统向用户发送的通知消息，包括消息类型、标题、正文、跳转链接、是否已读和创建时间。

#### 5.3.11 指导记录表 `guidance_record`

用于保存教师指导过程记录，包括指导时间、方式、问题、建议和后续跟进。

#### 5.3.12 评价记录表 `evaluation_record`

用于保存教师和学院的评价信息，包括阶段评语、总结评语、终评成绩、维度得分 JSON、优点、改进建议、学院评语、学院评分、提交标记和确认标记。

#### 5.3.13 学院入驻申请表 `college_application`

用于保存学院入驻平台的申请信息，包括学校名称、学院名称、联系人、联系电话、描述、审核状态和审核意见。

#### 5.3.14 审计日志表 `audit_log`

用于保存关键操作日志，包括日志类型、操作人、动作描述、详情和创建时间。

#### 5.3.15 系统设置表 `system_setting`

用于保存系统参数，如提醒阈值、预警等级、提醒模板等，可支持后台配置驱动。

### 5.4 索引设计建议

为保证查询性能，建议在以下字段建立索引。

1. `user_account.account`
2. `user_account.college_id`
3. `student.student_no`
4. `teacher.employee_no`
5. `mentor_application.student_id`
6. `mentor_application.teacher_id`
7. `internship_application.student_id`
8. `form_instance.student_id`
9. `form_instance.template_code`
10. `message_notice.user_id`
11. `audit_log.operator_id`
12. `system_setting.category`

### 5.5 数据完整性设计

1. 关键主键统一采用字符串 ID。
2. 账号、学号、工号等唯一业务标识需要唯一性校验。
3. 审核动作必须校验当前状态，防止非法状态跳转。
4. 学院级数据操作必须校验 `college_id`，防止跨学院访问。
5. JSON 字段在服务层完成结构校验，避免脏数据入库。

## 6. 接口详细设计

### 6.1 接口设计原则

1. 所有接口统一以 `/api` 作为根路径。
2. 除登录与健康检查外，接口统一使用令牌认证。
3. 返回结构统一为 `success + data + message`。
4. 写操作接口统一在服务层完成权限判断、业务校验和日志记录。

### 6.2 认证接口

1. `POST /auth/login`：账号密码登录，返回令牌及用户信息。
2. `GET /auth/me`：获取当前登录用户信息。
3. `POST /auth/change-password`：修改当前用户密码。
4. `GET /auth/login-records`：获取当前用户登录日志。

### 6.3 公共接口

1. `GET /health`：系统健康检查。
2. `GET /dashboard`：按角色返回首页统计看板。
3. `GET /messages`：获取当前用户消息列表。
4. `POST /messages/{id}/read`：标记消息为已读。

### 6.4 核心业务接口

1. 学院端学生管理接口：学生新增、查询、状态切换、密码重置。
2. 学院端教师管理接口：教师新增、查询、状态切换、密码重置。
3. 指导关系接口：指导申请发起、教师确认、学院复核。
4. 实习申请接口：学生提交、学院审批。
5. 表单接口：模板查询、表单新建、更新、教师审核、学院归档。
6. 指导记录接口：教师新增与查询指导记录。
7. 评价接口：教师保存评价、学院确认评价。
8. 管理接口：学院入驻审核、基础数据查询、系统参数维护、日志查询。

### 6.5 文件接口设计

文件管理接口负责附件上传和下载。

1. 上传时校验文件大小、扩展名和用户身份。
2. 返回附件访问路径与原始文件名。
3. 存储目录按业务日期和随机文件名组织，避免重名。
4. 附件元数据写入业务表 JSON 字段，便于业务记录引用。

## 7. 界面详细设计

### 7.1 界面设计原则

1. 统一门户风格，按角色加载不同菜单。
2. 以任务驱动为核心，首页优先展示待办、进度和风险。
3. 表单页采用“列表 + 编辑弹窗/详情页”方式，降低学习成本。
4. 审核页重点展示状态、时间、意见和可执行动作。
5. 全站状态标签颜色统一，便于用户识别流程位置。

### 7.2 页面结构设计

前端采用统一门户布局 `PortalLayout`，顶部展示系统标题与当前用户信息，左侧展示角色菜单，主区域按路由切换业务页面。

### 7.3 登录界面设计

登录页包含账号、密码输入框和登录按钮。登录成功后根据用户角色分别进入学生、教师、学院管理员、超级管理员门户首页。首次改密用户在进入业务前强制完成密码修改。

### 7.4 学生端界面设计

学生端包括以下页面。

1. 首页看板：显示指导关系状态、实习状态、待办数量、归档数量。
2. 指导教师申请页：展示申请记录列表、当前状态、申请按钮和备注输入区。
3. 实习申请页：展示申请表单、实习单位选择、起止日期、备注和附件上传区。
4. 表单中心页：按模板分类展示表单列表，支持新建、编辑、提交、查看历史。
5. 消息中心页：展示未读消息、已读消息及跳转入口。
6. 结果查看页：展示评价结果、归档结果和最终成绩。

### 7.5 指导教师端界面设计

教师端包括以下页面。

1. 首页看板：显示指导学生数量、待审核数量、归档数量、完成率。
2. 指导申请处理页：展示待确认申请、申请详情、通过与驳回操作。
3. 材料审核页：展示学生表单列表、表单详情、评分和审核意见输入区。
4. 指导记录页：展示指导记录列表和新增记录表单。
5. 评价管理页：展示学生评价列表和评价编辑区。
6. 预警提醒页：展示超时待办、风险等级和提醒按钮。
7. 消息中心页：展示系统通知和处理结果。

### 7.6 学院管理员端界面设计

学院端包括以下页面。

1. 首页看板：显示学生数、教师数、待审批数、待归档数和风险数。
2. 学生管理页：支持学生列表、新增、重置密码、启停用。
3. 教师管理页：支持教师列表、新增、重置密码、启停用。
4. 指导关系页：展示待复核关系和复核结果。
5. 实习单位管理页：支持单位列表、新增和维护。
6. 实习申请审批页：展示申请详情、审批意见、单位反馈和审批结果。
7. 归档中心页：展示待归档表单列表，支持单条或批量归档、退回。
8. 评价确认页：展示教师评价并执行确认操作。
9. 预警中心页：展示超时任务并支持发送提醒。
10. 报表中心页：展示学院维度统计数据。
11. 消息中心页：展示审批相关消息。

### 7.7 超级管理员端界面设计

超级管理员端包括以下页面。

1. 首页看板：显示活跃用户、学院申请数、待审核申请数、表单总量和未读消息数。
2. 基础数据页：维护平台基础档案与学院信息。
3. 参数配置页：维护提醒规则、阈值和模板。
4. 表单模板页：管理模板结构、适用类型、启停状态。
5. 日志审计页：查询登录日志、审批日志和系统操作日志。

## 8. 安全与权限设计

### 8.1 认证安全设计

1. 登录密码采用 SHA-256 摘要存储。
2. 登录成功后通过令牌维持会话状态。
3. 登录接口校验账户状态，禁用账户不允许登录。
4. 首次登录和密码重置后必须修改密码。

### 8.2 权限控制设计

1. 路由层按角色控制前端菜单和页面访问。
2. 后端接口层按角色和数据归属双重校验。
3. 学院管理员只能处理本学院数据。
4. 教师只能处理与自己存在有效指导关系的学生材料。
5. 学生只能查看和修改自己的业务数据。

### 8.3 数据安全设计

1. 关键业务操作写入审计日志。
2. 上传文件限制大小并进行类型校验。
3. 敏感配置不在日志中明文输出。
4. 数据库定期备份，附件目录增量备份。

## 9. 非功能设计

### 9.1 性能设计

1. 常用列表页采用分页和条件过滤设计。
2. 看板统计优先返回聚合结果，避免前端大批量计算。
3. JSON 扩展字段仅在详情场景读取完整内容。
4. 通过索引优化学生、教师、表单、消息等高频查询。

### 9.2 可靠性设计

1. 核心写操作采用事务控制，避免流程数据不一致。
2. 消息通知与日志写入纳入业务处理链路。
3. 文件上传失败时回滚业务引用，保证附件与记录一致。

### 9.3 可维护性设计

1. 前端按角色门户与通用布局拆分。
2. 后端按控制层、服务层、数据访问层分层组织。
3. 业务状态集中定义为枚举，减少散落常量。
4. 系统参数采用配置化存储，降低硬编码风险。

### 9.4 可扩展性设计

1. 表单模板采用 JSON 字段描述结构，可快速扩展新表单。
2. 系统参数表可扩展新的提醒规则和阈值。
3. 角色门户结构清晰，可继续增加校级管理员、企业导师等角色。
4. 文件服务与业务服务分离，便于后续迁移到对象存储。

## 10. 运行与部署设计

### 10.1 运行环境

1. 前端运行环境：Node.js 构建，Nginx 提供静态资源服务。
2. 后端运行环境：JDK 17 及以上，Spring Boot 单体应用。
3. 数据库环境：MySQL 8.x，开发联调可使用 H2。
4. 操作系统环境：Windows Server 或 Linux 均可。

### 10.2 部署步骤

1. 初始化数据库结构和基础数据。
2. 配置后端数据库连接、附件存储目录和运行参数。
3. 构建前端静态资源并发布。
4. 打包后端 Jar 并启动服务。
5. 配置反向代理、日志目录和备份策略。

### 10.3 日志与监控

1. 应用日志记录启动、异常、认证和关键业务信息。
2. 审计日志记录登录、审核、提醒、配置修改等行为。
3. 健康检查接口用于监控平台服务可用性。

## 11. 结论

本详细设计方案基于《需求规格说明书V2》和当前一期实现架构，完成了系统总体结构、模块划分、业务流程、数据库、接口、界面、安全和部署方案的细化设计。该方案能够支撑师范生教育实习全过程管理平台的一期建设与交付，并为后续二期扩展、流程细化和数据分析建设提供稳定基础。
